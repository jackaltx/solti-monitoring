## Configuration Architecture

### The Alloy Configuration File

**Single file**: `/etc/alloy/config.alloy` (generated from templates)

```
/etc/alloy/
├── config.alloy              # Main config (generated by role)
└── /var/lib/alloy/           # Data directory (positions, state)
```

### Configuration Structure

Alloy config is built from multiple template sections:

```hcl
// 1. Journal source (always included)
loki.source.journal "read" {
  // Reads systemd journal
}

// 2. Optional log sources (conditionally included)
local.file_match "apache_access" { ... }  // if alloy_monitor_apache
loki.source.file "fail2ban" { ... }       // if alloy_monitor_fail2ban
loki.source.journal "gitea" { ... }       // if alloy_monitor_gitea

// 3. Optional filters (conditionally included)
loki.process "filter_cron_noise" { ... }  // if alloy_filter_cron_noise

// 4. Loki output(s) (always included)
loki.write "endpoint_localhost" {
  endpoint {
    url = "http://127.0.0.1:3100/loki/api/v1/push"
  }
}
```

### How Role Variables Map to Configuration

```yaml
# In inventory:
alloy_loki_endpoints:
  - label: localhost
    endpoint: "127.0.0.1"
  - label: monitor11
    endpoint: "10.0.0.11"
  ↓
# Generates two loki.write blocks:
loki.write "endpoint_localhost" { ... }
loki.write "endpoint_monitor11" { ... }

alloy_monitor_apache: true
  ↓
# Includes Apache log collection templates:
local.file_match "apache_access" { ... }
loki.source.file "apache_access" { ... }
loki.process "apache_access" { ... }

alloy_filter_cron_noise: true
  ↓
# Adds cron noise filter to journal pipeline:
loki.process "filter_cron_noise" {
  stage.drop {
    expression = `.*CRON.*pam_unix.*session (opened|closed).*`
  }
}
```

### Key Configuration Variables

**Core Variables:**
```yaml
alloy_state: present                          # Install/remove control
alloy_config: /etc/alloy/config.alloy        # Config file path
alloy_custom_args: "--disable-reporting --server.http.listen-addr=0.0.0.0:12345"
```

**Loki Endpoints (REQUIRED):**
```yaml
alloy_loki_endpoints:
  - label: localhost      # Label used in config
    endpoint: "127.0.0.1" # Loki server IP/hostname (port 3100 assumed)
```

**Log Sources (all default to false):**
```yaml
alloy_monitor_apache: false           # Apache access/error logs
alloy_monitor_ispconfig: false        # ISPConfig logs
alloy_monitor_fail2ban: false         # Fail2ban file-based logs
alloy_monitor_fail2ban_journal: false # Fail2ban journald logs
alloy_monitor_mail: false             # Postfix/Dovecot logs
alloy_monitor_bind9: false            # Bind9 DNS logs
alloy_monitor_wg: false               # WireGuard VPN logs
alloy_monitor_gitea: false            # Gitea logs
alloy_monitor_ufw: false              # UFW firewall logs
```

**Filters:**
```yaml
alloy_filter_cron_noise: false  # Filter routine cron execution (keep errors)
```

**Permissions:**
```yaml
alloy_additional_groups: []  # Groups to add alloy user to for log access
# Example: ['git', 'syslog', 'adm']
```

## Test/Deploy Workflow

**IMPORTANT**: Always test Alloy config changes before deploying!

### Step 1: Test Configuration

```bash
# Renders config to /tmp and validates - does NOT restart service
cd mylab
ansible-playbook --become-password-file ~/.secrets/lavender.pass \
  ./playbooks/fleur/91-fleur-alloy-test.yml
```

**What Happens:**
1. Ansible renders config from templates
2. Writes to `/tmp/alloy-test-config-<timestamp>.alloy`
3. Runs `alloy fmt <file>` (format check)
4. Runs `alloy validate <file>` (syntax/semantic check)
5. Reports success or failure
6. **Service NOT touched**

**Check Test Output:**
```bash
# SSH to target host
ssh fleur

# View test config
ls -lt /tmp/alloy-test-config-* | head -1
cat /tmp/alloy-test-config-<timestamp>.alloy

# Check validation results in playbook output
```

### Step 2: Deploy Configuration

```bash
# ONLY run this after test passes!
cd mylab
ansible-playbook --become-password-file ~/.secrets/lavender.pass \
  ./playbooks/fleur/22-fleur-alloy.yml
```

**What Happens:**
1. Renders config from templates
2. Writes to `/etc/alloy/config.alloy`
3. Runs `alloy fmt` and `alloy validate`
4. **Restarts alloy service** (via handler)
5. Service picks up new config

**Verify Deployment:**
```bash
# Check service restarted successfully
systemctl status alloy

# Check for errors
journalctl -u alloy -n 50 --no-pager

# Verify new config loaded
curl -s http://127.0.0.1:12345/metrics | grep alloy_build_info
```

**What Gets Verified** ([molecule/proxmox/verify.yml](molecule/proxmox/verify.yml)):
- [ ] Alloy service running and enabled
- [ ] `/etc/alloy/config.alloy` exists, mode `0644`
- [ ] Config passes `alloy fmt` and `alloy validate`
- [ ] Metrics endpoint responding on port 12345
- [ ] No errors in service startup
- [ ] Git commit information captured
- [ ] Test report generated in `verify_output/<distro>/`

**Expected Labels:**

- `job`: "integrations/node_exporter"

```yaml

```

- Parsers for Apache log formats
- Labels: `log_type`, `status_code`, `method`, `uri`

**Requirements:**
- Apache installed with logs in `/var/log/apache2/` or `/var/log/httpd/`
- Alloy user needs read access (group `adm` or `syslog`)

**Verification:**

**Troubleshooting:**

## Filtering Features

### Cron Noise Filter

```yaml
alloy_filter_cron_noise: true
```

**Drops:**
- PAM session opened/closed for cron
- Routine CMD execution (ISPConfig, getmail, system cron)
- Successful cron completions

**Keeps:**
- Errors in cron execution
- Failed authentication
- Command failures

**Verification:**
```bash
# Check filter in config
sudo grep -A 20 'loki.process "filter_cron_noise"' /etc/alloy/config.alloy

# Query Loki - should NOT see routine cron noise
curl -G "http://loki.example.com:3100/loki/api/v1/query" \
  --data-urlencode 'query={hostname="yourhost"} |= "CRON"' \
  --data-urlencode 'limit=50'
# Should see only errors, not routine executions
```

## Common Errors

### "dial tcp ... connection refused" (Loki unreachable)

**Cause:** Loki endpoint not accessible

**Fix:**
```bash
# Check Loki is running
ssh monitor11 systemctl status loki

# Check network connectivity
ping 10.0.0.11
telnet 10.0.0.11 3100

# Check firewall
sudo firewall-cmd --list-all
```

### "permission denied" reading log files

**Cause:** Alloy user lacks read access to log files

**Fix:**
```bash
# Check file permissions
ls -l /var/log/apache2/access.log

# Add alloy to required group
sudo usermod -aG adm alloy

# Or set in inventory:
alloy_additional_groups: ['adm', 'git']

# Redeploy
cd mylab
./manage-svc.sh alloy deploy
```

### "component X is not defined"

**Cause:** Config references component before it's defined

**Fix:**

## Notes

- **Test before deploy**: Always use test playbook before deploying to production
- **Config is monolithic**: One file, not directory-based like Telegraf
- **Live reload**: Not yet supported - requires service restart
- **Log file access**: Alloy user must have read permissions (via groups)
- **Filtering reduces noise**: Use filters to keep Loki storage manageable
- **Multiple endpoints**: Logs can be sent to multiple Loki instances simultaneously
- **Metrics for observability**: Alloy exposes Prometheus metrics about itself
- **Journal vs file sources**: Journal is more reliable, file-based for legacy apps
